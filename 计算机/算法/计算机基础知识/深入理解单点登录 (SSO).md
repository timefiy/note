假设你在一家公司工作，每天要用到很多不同的系统：

- 企业邮箱 `mail.company.com`
- 代码仓库 `git.company.com`
- 项目管理工具 `pm.company.com`
- 文档系统 `docs.company.com`
- 人事系统 `hr.company.com`

你可以使用同一个公司账号（比如带有员工 ID 的邮箱账号 `tom@internal.company.com` ）登录所有系统，但由于每个系统使用不同的子域名，你需要在每个系统重复登录。

我们在 [Session 和 Cookie](https://labuladong.online/zh/algo/other-skills/session-and-cookie/) 中讲过，当你登录 `git.company.com` 后，服务器会设置一个 Session Cookie：

这个 Cookie 只对 `git.company.com` 有效，当你访问 `mail.company.com` 时，浏览器不会发送这个 Cookie，所以 `mail.company.com` 不知道你已经登录过了，要求你重新登录。

那么最简单的一个方案就是，让 `git.company.com` 的服务器直接对主域名 `company.com` 设置 Cookie，这样所有 `*.company.com` 子域名都能读取这个 Cookie，不就实现"共享登录状态"了吗？

技术上确实可以，但这种做法有严重的安全隐患和架构问题：

**1、安全问题：一个服务的漏洞会危及所有服务** 。

如果 `git.company.com` 设置了 `domain=.company.com` 的 Session Cookie，那么所有子域名的服务（ `mail.company.com` 、 `hr.company.com` 等）都能读取和修改这个 Cookie。

假设 `mail.company.com` 存在安全漏洞漏洞，攻击者可以通过这个漏洞窃取 Session Cookie，那么就可以冒充用户访问所有其他系统（邮箱、代码仓库、人事系统等）。

**2、架构问题：Session 的数据结构需要统一**

共享 Cookie 意味着共享 Session ID，但不同服务需要访问同一个 Session 数据，这要求：

- 所有服务共享同一个 Session 存储（比如 Redis）
- 所有服务的 Session 数据结构必须兼容
- 不同团队开发的服务需要协调 Session 字段

这显现是不现实的，每个团队的业务不同，Session 中记录的数据结构也不同，强行共享会导致紧耦合，任何一个服务修改 Session 结构都可能影响其他服务。

**3、无法集成第三方服务**

如果你的公司想要集成一些第三方 SaaS 服务，这些服务有自己的域名（如 `yourcompany.slack.com` ），无法通过这种简单的共享 Cookie 方案来实现。

解决这个问题的现代化方案是使用 **SSO (Single Sign-On，单点登录)** 。

SSO 通过引入一个独立的认证中心（比如 `op.company.com` ）来解决前面提到的问题：

- **安全隔离** ：每个子系统有自己的 Session，一个子系统被攻击不会影响其他子系统。
- **跨域名支持** ：可以集成不同域名的服务，包括第三方 SaaS。
- **统一的认证逻辑** ：所有子系统只需专注业务逻辑，不需要各自实现用户登录逻辑。
- **统一的账号管理** ：用户只需要在认证中心登录一次，就可以访问所有相关的子系统，无需重复登录。

## SSO 的工作原理

要实现 SSO，核心是要有一个 **统一的认证中心 (Authentication Server)** ，所有应用都信任这个认证中心。

架构图大概是这样的：

```
认证中心

              ↓ 信任关系

      ┌───────┼───────┬───────┐

      ↓       ↓       ↓       ↓

   应用 A    应用 B   应用 C  应用 D
```

涉及的主要角色有三个：

1. **认证中心 (Authentication Server / Identity Provider)**
	- 统一管理用户的账号和密码
	- 负责验证用户身份（检查用户名密码是否正确）
	- 为用户颁发"身份凭证"
2. **应用 (Application / Service Provider)**
	- 各种业务系统，比如邮箱、CRM、文档系统
	- 信任认证中心
	- 通过验证"身份凭证"来确认用户身份
3. **用户 (User)**
	- 想要访问多个应用
	- 只需要在认证中心登录一次

## OIDC 实现 SSO

在 [OAuth 2.0 和 OIDC 认证](https://labuladong.online/zh/algo/computer-science/oidc/) 这篇文章中，我们详细介绍了 OIDC 如何实现「用 Google 账号登录」这种第三方登录。

OIDC 不仅可以做第三方登录，也是实现 SSO 的绝佳选择，而且是目前最推荐的现代 SSO 方案。

简单回顾一下 OIDC 的核心概念：

- **OP** (OpenID Provider)：认证服务器，负责验证用户身份（相当于 SSO 中的认证中心）。
- **RP** (Relying Party)：依赖方，就是各个应用。
- **ID Token** ：用户身份信息，JWT 格式的令牌。

在单应用场景下，用户登录一个应用，OP 给这个应用颁发 ID Token，应用验证 Token 后允许用户访问。

在 SSO 场景下，区别在于：

**OP 本身也维护一个 Session，当用户访问多个应用时，OP 的 Session 依然有效，所以 OP 可以直接给每个应用颁发 Token，无需用户重新登录** 。

### OIDC SSO 流程

假设公司使用 OIDC 实现 SSO，有一个统一的 OP（比如公司的认证服务器），还有应用 A（邮箱）和应用 B（CRM）。

**第一次访问应用 A，和标准的 OIDC 流程非常类似** ：

1. 用户访问应用 A，域名为 `mail.company.com` 。
2. 应用 A 发现你未登录，重定向到 OP 的 `/authorize` 端点：

```
https://op.company.com/authorize?

  response_type=code&

  client_id=mail_client_id&

  redirect_uri=https://mail.company.com/callback&

  scope=openid,email,profile&

  nonce=nonce456
```

1. OP 检查用户浏览器在 `op.company.com` 域名下的 Session Cookie，发现没有 Session（用户还没登录）。
2. OP 显示登录页面，要求用户输入用户名和密码。
3. OP 验证凭据成功后，在后端缓存中创建一个 Session，并在用户浏览器设置 Session Cookie。 **注意这是在 `op.company.com` 域名下设置的 Cookie** 。
4. OP 重定向回应用 A，URL 中带上 Authorization Code：

```
https://mail.company.com/callback?code=AUTH_CODE_123
```

1. 应用 A 的后端用 Authorization Code 换取 Token（调用 OP 的 `/token` 端点）：

```
POST https://op.company.com/token

Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&

code=AUTH_CODE_123&

client_id=mail_client_id&

client_secret=mail_secret&

redirect_uri=https://mail.company.com/callback
```

1. OP 返回 ID Token 和 Access Token：

```
{

  "access_token": "ACCESS_TOKEN_XYZ",

  "id_token": "ID_TOKEN_ABC...",

  "token_type": "Bearer",

  "expires_in": 3600

}
```

1. 应用 A 验证 ID Token 的签名和 nonce，确认用户身份。
2. 用户成功登录应用 A，应用 A 的后端服务器在 `mail.company.com` 域名下设置了一个 Session Cookie，记录了用户的登录状态。

**第二次访问应用 B（SSO 发生）** ：

1. 用户打开新标签页，访问应用 B（CRM 系统），域名为 `crm.company.com` 。
2. 应用 B 发现用户未登录，重定向到 OP 的 `/authorize` 端点：

```
https://op.company.com/authorize?

  response_type=code&

  client_id=crm_client_id&

  redirect_uri=https://crm.company.com/callback&

  scope=openid,email,profile&

  nonce=nonce012
```

1. **OP 检查用户浏览器在 `op.company.com` 域名下的 Session Cookie，发现 Session 依然有效（刚才登录过了）** 。
2. **OP 直接颁发新的 Authorization Code，不需要用户再输入密码** 。
3. OP 重定向回应用 B：

```
https://crm.company.com/callback?code=AUTH_CODE_456
```

1. 应用 B 用 Authorization Code 换取 Token。
2. OP 返回 ID Token 和 Access Token（ID token 中的 `aud` 是 `crm_client_id` ，和应用 A 的 Token 不同）。
3. 应用 B 验证 ID Token 的签名和 nonce，确认用户身份。
4. 用户成功登录应用 B，应用 B 的后端服务器在 `crm.company.com` 域名下设置了一个 Session Cookie，记录了用户的登录状态。

### 关键点

注意第 3-4 步，这就是 OIDC SSO 的关键：

**当用户访问不同的应用时，每个应用都会重定向到 OP，OP 通过 `op.company.com` 域名下的 Session Cookie 识别用户，直接颁发新的 Authorization Code，不需要用户重复进行登录** 。

虽然多个应用共享同一个 OP Session，但每个应用获得的 Token 是独立的。

每个应用的 ID Token 中的 `aud` (audience) 不同，比如应用 A 的 ID Token： `"aud": "mail_client_id"` ，应用 B 的 ID Token： `"aud": "crm_client_id"` 。同时，每个应用的 token 过期时间和权限 scope 也可以不同。

这种设计保证了安全性：即使黑客拿到了应用 A 的 Token，也无法用这个 Token 访问应用 B。

## OIDC SSO 登出

SSO 的登出稍微有些复杂，因为要清理多处 Session 才能完成登出。

比方说一个用户通过 OP 认证服务器登录了多个应用，那么就会存在以下多个活跃的 Session：

- `op.company.com` 认证服务器的 Session。
- `mail.company.com` 邮件应用的 Session。
- `crm.company.com` CRM 应用的 Session。
- ...

当用户在 `mail.company.com` 应用中点击登出按钮时，会发生什么呢？有两种情况：

- RP-Initiated Logout（RP 发起的登出），即仅退出当前应用的登录，其他应用（如 `crm.company.com` ）的登录状态不受影响。
- Single Logout（单点登出），即退出所有应用的登录，包括当前应用和所有其他应用。

### RP-Initiated Logout

假设用户在 `mail.company.com` 应用中触发 RP-Initiated Logout，那么流程如下：

1. `mail.company.com` 应用清除自己的 Session。
2. `mail.company.com` 应用将用户重定向到 OP 的 `/end_session` 端点：

```
https://op.company.com/end_session?id_token_hint=ID_TOKEN_ABC
```

1. OP 收到请求后，清除 `op.company.com` 的 Session。

上述步骤中，清除 `op.company.com` 的 Session 是必须的，否则用户再次访问 `mail.company.com` 应用时，会重定向到 `op.company.com/authorize` 页面进行 SSO 登录，而由于 `op.company.com` 的 Session 还存在，用户会直接登录成功。那么登出就失去了意义。

### Single Logout

假设用户在 `mail.company.com` 应用中触发 Single Logout，那么流程如下：

1. `mail.company.com` 应用清除自己的 Session。
2. `mail.company.com` 应用通知 OP：用户要执行单点登出，请通知所有曾经登录过的应用登出。
3. OP 清除 `op.company.com` 的 Session。
4. **OP 向所有曾经登录过的应用发送登出通知** （通过 Back-Channel Logout 机制）。
5. `crm.company.com` 等其他应用收到通知后，清除各自的 Session。

这样，用户在一个应用中登出，所有应用的登录状态都会失效，实现真正的「单点登出」。

**Back-Channel Logout 的技术实现** ：

OP 需要记录用户登录过哪些应用，当收到单点登出请求时，OP 会向每个应用的后端发送 POST 请求：

```
POST https://<service_name>.company.com/backchannel-logout

LOGOUT_TOKEN: <logout_token>
```

`logout_token` 一般是一个 JWT，包含用户标识和登出事件信息。

应用收到这个请求后，验证 `logout_token` 的签名确实由 OP 颁发，然后清除对应用户的 Session。

这样，用户在一个应用中登出，所有应用的登录状态都会失效，实现真正的「单点登出」。
